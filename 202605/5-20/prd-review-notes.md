# PRD 评审备注

评审对象：`D:\学习\czy0821.github.io\202605\5-20\ai-task-flow-prd-v0yf-修改前.md`

原型地址：https://chengzi3321.github.io/New-project-6/

## 总体结论

这份 PRD 已经覆盖了任务列表页的主要页面结构，包括顶部状态、指标区、状态筛选、搜索、任务列表、任务详情和操作入口。但如果要交给开发继续实现，还有一些需求边界不够清楚，容易导致开发时自行猜测。

## 不清楚的需求点

### 1. “新建任务流”到底是否属于 v0 范围不清楚

原型右上角有「新建任务流」按钮，但 PRD 的“不在本次范围”中写到不扩展到任务创建页，同时功能列表又提到系统需要具备 `Create` 能力。

需要明确：

- v0 是否只展示「新建任务流」按钮？
- 点击后是无交互、弹提示、打开表单，还是进入新页面？
- 如果要创建任务，表单字段、必填项、默认状态、创建成功后的列表刷新规则是什么？

### 2. “编辑”和“催办”的操作结果不清楚

原型中的每条任务都有「查看」「编辑」「催办」按钮，PRD 也要求保留入口，但没有说明点击后的具体行为。

需要明确：

- 点击「编辑」后打开弹窗、跳转页面，还是只预留入口？
- 哪些字段允许编辑？
- 点击「催办」后是发送通知、更新任务状态、写入日志，还是只展示提示？
- 操作成功或失败时页面如何反馈？

### 3. 数据来源和 API 范围不清楚

PRD 一方面说 v0 聚焦任务列表页，不展开接口协议细节；另一方面验收标准又要求字段、指标、系统状态必须由后端 API 提供，并要求支持 `Create`、`Read`、`Update`、`Delete/Archive`。

需要明确：

- v0 是继续使用前端 mock 数据，还是必须接真实 API？
- 如果接 API，需要哪些接口：列表、详情、创建、更新、删除/归档、催办？
- `deliveryRate`、`systemStatusText`、`systemStatusCode` 等字段由哪个接口返回？
- 接口失败时页面如何展示？

### 4. 状态规则不够清楚

原型支持 `running`、`review`、`blocked`、`done` 四种状态，筛选还包含 `all`。PRD 说明了状态筛选值，但没有定义状态含义和状态流转规则。

需要明确：

- `running`、`review`、`blocked`、`done` 分别代表什么业务阶段？
- 哪些状态可以切换到哪些状态？
- `blocked` 是否算告警任务？
- `done` 是否还能编辑、催办或归档？
- 状态变化是否需要记录操作日志？

### 5. “按优先级自动排序”的规则不清楚

原型任务列表标题旁写着“按优先级自动排序”，任务数据中有 `P0`、`P1`、`P2`，但 PRD 没有写清楚排序规则。

需要明确：

- 默认排序是 `P0 > P1 > P2` 吗？
- 同优先级任务按创建时间、截止时间还是进度排序？
- 搜索和筛选后是否仍保持优先级排序？

### 6. 顶部指标口径不够清楚

PRD 写了活跃任务数、待处理告警数、今日交付率，但只有部分计算规则。

需要明确：

- “今日交付率”的计算周期是自然日、工作日，还是后端指定时间范围？
- “今日”按哪个时区计算？
- `deliveryRate=82` 是百分比数字、字符串，还是小数 `0.82`？
- `activeTaskCount` 是否包含 `blocked` 和 `review`？

### 7. 字段命名与原型代码不完全一致

PRD 中使用了 `ownerName`、`riskLevel`、`nextAction`，但原型代码中对应字段是 `owner`、`risk`、`next`。

需要明确：

- 后续开发以 PRD 字段命名为准，还是以原型代码字段为准？
- 如果接 API，接口字段是否需要统一为 `ownerName`、`riskLevel`、`nextAction`？
- 前端是否需要做字段映射？

### 8. 删除/归档能力是否进入 v0 不清楚

PRD 的验收标准中写到系统必须支持 `Delete/Archive`，但原型页面没有删除或归档入口，非目标里也没有明确说明归档是否本期实现。

需要明确：

- v0 是否需要前台删除或归档入口？
- 如果需要，入口放在哪里？
- 删除和归档是同一个动作还是两个动作？
- 删除或归档后任务是否从默认列表中消失？

### 9. 操作日志没有明确要求

PRD 提到了查看、编辑、催办、CRUD 和状态数据，但没有说明是否需要记录操作日志。对于任务流系统来说，后续追踪“谁在什么时候做了什么”会比较重要。

需要明确：

- v0 是否需要操作日志？
- 哪些动作需要写日志：创建、编辑、催办、状态变化、删除/归档？
- 日志展示在任务详情区，还是暂时只在后端保存？
- 日志字段包括哪些：操作人、操作时间、操作类型、操作说明？

## 建议优先补充

建议先补清楚以下 3 个点，再进入开发：

1. v0 是否接真实 API，还是继续使用 mock 数据。
2. 「新建任务流」「编辑」「催办」三个按钮的点击结果。
3. 任务状态定义、状态流转和排序规则。
