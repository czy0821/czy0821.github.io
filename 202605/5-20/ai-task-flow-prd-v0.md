# PRD：AI 任务流管理系统任务列表页

## 1. 背景

随着团队开始使用 AI Agent 处理销售、客服、运营、研究等任务，任务数量会逐渐变多。如果没有统一的任务管理页面，用户很难快速知道：

- 当前有哪些 AI 任务正在执行
- 哪些任务需要人工确认
- 哪些任务已经完成
- 哪些任务执行失败或出现异常
- 每个任务由谁负责
- 任务最近有没有更新

因此需要一个“AI 任务流管理系统”的任务列表页，帮助用户集中查看和管理 AI 任务的执行情况。

当前已有一个前端原型，能够展示任务列表、状态、负责人、创建时间、更新时间、操作按钮和筛选功能。该原型目前主要用于确认页面感觉和核心流程，数据仍为假数据，尚未连接真实 API 和数据库。

## 2. 目标用户

### 主要用户

1. **业务负责人**

   需要快速了解 AI 任务整体进展，比如哪些任务正在执行、哪些任务异常、哪些任务需要人工确认。

2. **运营人员**

   需要每天查看任务状态，处理异常任务，跟进待确认任务。

3. **任务负责人**

   需要查看自己负责的 AI 任务，确认任务结果，必要时编辑、重试或归档任务。

### 次要用户

1. **产品经理**

   用于观察 AI 任务流程是否清晰，后续优化任务状态和操作规则。

2. **技术/开发人员**

   用于排查异常任务，查看未来接入的运行日志和执行记录。

## 3. 核心场景

### 场景 1：查看所有 AI 任务

用户进入任务列表页后，可以看到当前所有 AI 任务，包括任务名称、负责人、状态、创建时间、更新时间、智能体和操作按钮。

### 场景 2：快速判断任务整体情况

用户通过顶部统计区域快速了解：

- 有多少任务正在进行中
- 有多少任务待人工确认
- 今天完成了多少任务
- 有多少异常任务

### 场景 3：筛选需要处理的任务

用户可以通过状态筛选快速找到：

- 进行中的任务
- 待确认的任务
- 已完成的任务
- 异常任务

也可以通过搜索框按任务名称、负责人或智能体查找任务。

### 场景 4：查看任务详情

用户点击某个任务后，右侧显示任务详情，包括任务描述、执行步骤和备注，帮助用户判断这个任务当前进行到哪一步。

### 场景 5：处理任务

用户可以对任务进行操作：

- 查看
- 编辑
- 标记待确认
- 标记完成
- 归档
- 重试异常任务

当前原型里这些操作大多是演示效果。未来真系统中，操作结果需要保存到数据库。

## 4. 功能列表

### 4.1 任务概览

页面顶部展示任务统计：

- 进行中数量
- 待人工确认数量
- 今日完成数量
- 异常任务数量

要求：

- 统计数字应基于真实任务数据计算
- 状态变化后，统计数字应同步更新

### 4.2 任务列表

任务列表需要展示以下字段：

- 任务名称
- 任务描述
- 状态
- 优先级
- 负责人
- 智能体
- 创建时间
- 更新时间
- 操作按钮

要求：

- 任务名称必须清晰可读
- 状态需要用明显标签区分
- 异常任务需要容易被发现
- 创建时间和更新时间都要展示

### 4.3 搜索

用户可以通过搜索框搜索任务。

支持搜索范围：

- 任务名称
- 负责人
- 智能体

要求：

- 输入关键词后，任务列表应显示匹配结果
- 没有匹配结果时，应显示空状态提示

### 4.4 状态筛选

用户可以按状态筛选任务。

状态包括：

- 全部状态
- 进行中
- 待确认
- 已完成
- 异常

要求：

- 选择状态后，列表只展示对应状态的任务
- 状态筛选应和搜索条件同时生效

### 4.5 优先级筛选

用户可以按优先级筛选任务。

优先级包括：

- 全部优先级
- 高优先级
- 中优先级
- 低优先级

要求：

- 选择优先级后，列表只展示对应优先级的任务
- 优先级筛选应和搜索、状态筛选同时生效

### 4.6 任务详情

用户点击任务后，右侧显示任务详情。

详情内容包括：

- 任务标题
- 任务描述
- 当前执行步骤
- 备注
- 标记待确认按钮
- 标记完成按钮

要求：

- 点击不同任务时，右侧详情应同步切换
- 任务步骤应能展示每一步的执行状态

### 4.7 任务操作

每条任务应提供操作按钮。

基础操作包括：

- 查看
- 编辑
- 归档

异常任务额外支持：

- 重试

要求：

- 点击查看，应进入或展示任务详情
- 点击编辑，应允许修改任务基础信息
- 点击归档，应将任务从默认列表中移出或标记为已归档
- 点击重试，应重新触发异常任务执行

### 4.8 新建任务

页面提供“新建任务”按钮。

未来应支持创建新 AI 任务。

基础字段建议包括：

- 任务名称
- 任务描述
- 负责人
- 智能体
- 优先级
- 输入内容

要求：

- 新建成功后，任务应出现在任务列表中
- 刷新页面后，新任务仍然存在

## 5. 非目标

本阶段暂不做以下内容：

- 不做完整 AI Agent 编排系统
- 不做复杂权限系统
- 不做多团队组织架构管理
- 不做完整运行日志详情页
- 不做复杂报表分析
- 不做移动端深度适配
- 不做真实 AI 执行引擎
- 不做消息通知系统

本阶段重点是：

> 先把任务列表页的核心信息、基础筛选和基础操作定义清楚。

## 6. 验收标准

### 6.1 页面字段验收

任务列表页必须包含：

- 任务名称
- 负责人
- 状态
- 创建时间
- 更新时间
- 操作按钮
- 状态筛选

### 6.2 列表展示验收

- 用户进入页面后，可以看到任务列表
- 每条任务展示完整基础字段
- 不同状态有明显区分
- 异常任务可以被快速识别

### 6.3 搜索和筛选验收

- 输入任务名称关键词，可以筛选出匹配任务
- 输入负责人关键词，可以筛选出匹配任务
- 输入智能体关键词，可以筛选出匹配任务
- 选择状态后，只展示对应状态任务
- 选择优先级后，只展示对应优先级任务
- 搜索、状态筛选、优先级筛选可以同时生效
- 无匹配结果时，页面显示空状态提示

### 6.4 详情查看验收

- 点击任务后，右侧显示对应任务详情
- 切换不同任务时，详情内容同步变化
- 详情中可以看到任务描述、执行步骤和备注

### 6.5 操作按钮验收

- 每条任务都有操作按钮
- 普通任务显示查看、编辑、归档
- 异常任务显示查看、编辑、重试
- 点击操作按钮时，不应误触发行点击逻辑
- 当前原型阶段，按钮可以先弹出演示提示

### 6.6 状态更新验收

- 点击“标记待确认”后，当前任务状态变为待确认
- 点击“标记完成”后，当前任务状态变为已完成
- 顶部统计数字随状态变化同步更新

原型阶段允许刷新后数据恢复。

真系统阶段要求刷新后状态仍然保留。

### 6.7 真系统验收

如果进入正式开发阶段，必须满足：

- 任务数据来自后端 API
- 新建、编辑、状态更新、归档、重试等操作调用后端 API
- 数据保存到数据库
- 刷新页面后数据不丢失
- 接口失败时有错误提示
- 用户只能看到自己有权限查看的任务

## 7. 当前原型说明

当前原型是一个前端可点击版本，主要用于确认产品方向。

当前原型中：

- 任务数据是假数据
- 统计数字基于假数据计算
- 操作按钮多数为演示效果
- 状态修改只在当前页面临时生效
- 没有真实 API
- 没有数据库

因此，当前原型不能直接上线。

它的价值是：

> 帮助团队确认任务列表页的结构、信息展示、筛选方式和基础操作是否合理。
