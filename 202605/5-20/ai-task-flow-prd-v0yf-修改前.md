# AI Task Flow PRD v0

## 背景
当前“AI 任务流管理系统”的任务列表页原型已经验证了基础信息架构，包括任务列表、状态筛选、顶部指标、任务详情和基础操作入口。但现阶段页面中的任务数据、指标数据和系统状态均为前端写死的假数据，无法支持真实业务使用。

因此，v0 版本的目标不是一次性做完整系统，而是在保持当前页面结构稳定的前提下，明确任务列表页最小可交付范围，让后续开发能够围绕清晰的数据字段、页面展示和业务规则逐步落地，避免范围失控和逻辑混乱。

## 目标用户
- 任务执行者：查看自己负责的任务、当前状态、截止时间和下一步动作。
- 任务管理者：查看活跃任务数量、告警任务数量、任务分布和整体交付进度。
- 审核/协作角色：查看任务详情、流程节点和当前处理情况。

## 用户故事
- 作为一个任务执行者，我希望进入页面后就能看到任务名称、负责人、状态、创建时间和截止时间，这样我可以快速判断先处理哪一项任务。
- 作为一个任务执行者，我希望按状态筛选任务，这样我可以只看当前需要处理的任务，而不用在全部任务中逐条查找。
- 作为一个任务执行者，我希望通过关键词搜索任务，这样我可以更快定位目标任务，而不是依赖手动浏览整个列表。
- 作为一个任务执行者，我希望点击一条任务后能在右侧看到详情，这样我可以在不离开列表页的情况下理解任务背景和流程节点。
- 作为一个任务管理者，我希望看到活跃任务数量、待处理告警数量和交付率，这样我可以快速判断当前任务系统的运行情况。
- 作为一个任务管理者，我希望看到 AI 运行状态，这样我可以判断当前系统是否稳定、是否需要关注异常。
- 作为一个审核或协作角色，我希望看到任务的负责人、风险等级、流程节点和下一步建议，这样我可以更快决定是否需要介入处理。
- 作为一个任务管理者，我希望为每条任务保留查看、编辑、催办入口，这样我可以对任务进行后续管理动作。
- 作为一个任务管理者，我希望系统支持任务的新建、读取、更新和删除或归档，这样任务列表页展示的数据能够来自真实业务而不是前端写死数据。

## 核心场景
1. 用户进入任务列表页后，需要快速看到当前有哪些任务正在处理、哪些任务已经阻塞、哪些任务待审核。
2. 用户需要按状态筛选任务，只查看“执行中”“待审核”“阻塞”或“已完成”的任务。
3. 用户需要点击某条任务，在同一页面右侧查看该任务的负责人、创建时间、截止时间、风险等级和流程节点。
4. 用户需要看到顶部指标，快速了解当前活跃任务数、待处理告警数和交付率。
5. 用户需要通过“查看”“编辑”“催办”这些操作入口，进入后续真实业务动作。

## 范围

### 本次范围（In Scope）
- 仅覆盖“任务列表页 v0”这一个页面，不扩展为完整任务流系统。
- 页面结构包含：顶部信息区、顶部指标区、状态筛选区、搜索区、任务列表区、任务详情区。
- 任务列表页展示的核心字段包含：
  - `title`
  - `ownerName`
  - `status`
  - `createdAt`
  - `deadline`
  - `priority`
  - `tags`
  - `progress`
  - `summary`
- 任务详情区展示的核心字段包含：
  - `title`
  - `status`
  - `ownerName`
  - `createdAt`
  - `deadline`
  - `riskLevel`
  - `summary`
  - `steps`
  - `nextAction`
- 页面支持的交互仅包括：
  - 状态筛选
  - 关键词搜索
  - 任务选中与详情切换
  - 查看、编辑、催办入口展示
- 页面需要为后续接入 API 预留真实字段和业务规则，但 v0 文档范围聚焦于列表页本身。

### 不在本次范围（Out of Scope）
- 不扩展到任务创建页、任务编辑页、任务详情独立页、看板页、报表页。
- 不定义复杂审批流、节点配置流、模型调度流。
- 不要求在本 PRD 中展开数据库表设计、接口协议细节、权限矩阵细节。
- 不在本次范围中增加与任务列表页无直接关系的新模块。

### 1. 任务列表页基础布局
- 功能说明：页面需要包含顶部信息区、指标区、状态筛选区、任务列表区、任务详情区。
- 对应页面：[index.html](</C:/Users/SMI/Documents/New project 6/index.html>)
- 数据字段：无
- 业务规则：该功能仅定义页面容器结构，不承载业务计算。

### 2. 顶部系统状态展示
- 功能说明：页面顶部展示系统状态文案，用于表达 AI 运行状态。
- 对应页面：[index.html](</C:/Users/SMI/Documents/New project 6/index.html>)

  - `systemStatusText`
  - `systemStatusCode`
- 业务规则：
  - v0 仅展示一个系统状态入口。
  - 后续系统状态由 API 返回。
  - 状态值至少支持：`stable`、`warning`、`error`。

### 3. 顶部指标展示
- 功能说明：页面展示活跃任务数量、待处理告警数量、今日交付率。
- 对应页面：[index.html](</C:/Users/SMI/Documents/New project 6/index.html>)
- 数据字段：
  - `activeTaskCount`
  - `alertTaskCount`

- 业务规则：
  - `activeTaskCount` = 状态不为 `done` 的任务数量。
  - `alertTaskCount` = 状态为 `blocked` 的任务数量。
  - `deliveryRate` 由后端计算并返回百分比数字。
  - 页面只负责展示，不在前端自行定义业务口径。

### 4. 状态筛选
- 功能说明：用户可以按状态筛选任务。
- 对应页面：[index.html](</C:/Users/SMI/Documents/New project 6/index.html>)

- 业务规则：
  - 支持的状态值固定为：`all`、`running`、`review`、`blocked`、`done`。
  - 任何任务必须且只能属于其中一个业务状态。
  - 当前筛选条件变化后，任务列表需要同步刷新。

### 5. 任务列表展示

## 非目标
- 不实现任务看板页、日历页、统计报表页等额外页面。
- 不实现复杂权限系统，不区分细粒度角色权限。
- 不实现评论、附件、@提醒、协同编辑等扩展协作功能。
- 不实现 AI 节点重试、节点配置编辑、模型选择等高级工作流配置能力。
- 不实现批量操作、批量导入导出。
- 不实现自定义筛选器、自定义字段、自定义状态流转。
- 不实现移动端专属交互方案，只要求基础响应式可读。
- 不实现完整消息中心，仅保留“催办”操作入口定义。

## 验收标准
- 页面必须包含 5 个明确区域：顶部信息区、指标区、状态筛选区、任务列表区、任务详情区。
- 页面顶部必须包含系统状态展示区域；当 `systemStatusCode` 为 `stable`、`warning`、`error` 时，页面必须展示对应的 `systemStatusText`。
- 每条任务在列表中必须展示以下字段：
  - `title`
  - `ownerName`
  - `status`
  - `createdAt`
  - `deadline`
  - 操作按钮
- 顶部指标必须有明确字段来源：
  - `activeTaskCount`
  - `alertTaskCount`
  - `deliveryRate`
- 当任务数据中有 3 条状态不为 `done` 的任务时，页面展示的 `activeTaskCount` 必须为 3。
- 当任务数据中有 2 条状态为 `blocked` 的任务时，页面展示的 `alertTaskCount` 必须为 2。
- 当后端返回 `deliveryRate=82` 时，页面必须展示 `82%`。
- 状态筛选只支持以下固定值：
  - `all`
  - `running`
  - `review`
  - `blocked`
  - `done`
- 当用户点击某个状态筛选项时，列表必须只展示符合该状态的任务；当用户点击 `all` 时，列表必须恢复展示全部任务。
- 当前生效的状态筛选项必须有明确选中态。
- 当用户输入关键词时，列表必须基于 `title`、`ownerName`、`tags`、`summary` 执行模糊匹配。
- 当搜索关键词与状态筛选同时存在时，列表结果必须同时满足这两个条件。
- 当搜索无结果时，页面必须展示空状态提示。
- 用户点击某条任务后，详情区必须同步展示该任务的：
  - `title`
  - `status`
  - `ownerName`
  - `createdAt`
  - `deadline`
  - `riskLevel`
  - `summary`
  - `steps`
  - `nextAction`
- 页面首次加载后，默认必须选中第一条任务，并在详情区展示该任务信息。
- 详情区中的流程节点必须按 `steps` 数组顺序展示，并且每个节点至少展示 `name` 和 `state`。
- 每条任务必须能对应唯一 `id`，用于查看、编辑、催办等操作。
- “查看”“编辑”“催办”必须作为独立操作入口存在，不能合并为模糊按钮。
- 当用户点击“查看”时，当前任务必须成为选中任务，且详情区必须同步切换。
- 当用户点击“编辑”或“催办”时，页面必须保留该操作入口，并能够基于当前任务 `id` 发起后续操作，不得丢失当前任务上下文。
- 任务列表页所需字段、指标字段、系统状态字段必须可由后端 API 提供，前端不再依赖写死假数据作为正式运行方案。
- 系统必须支持任务 `Create`、`Read`、`Update`、`Delete/Archive` 的基础能力：
  - `Create`：创建任务时，至少接收 `title`、`ownerId`、`status`、`deadline`、`priority`，并返回唯一 `id`。
  - `Read`：列表接口必须返回任务列表所需字段；详情接口必须返回详情区所需字段。
  - `Update`：更新任务后，再次读取任务时必须返回最新字段值和最新状态。
  - `Delete/Archive`：删除或归档后的任务不得继续出现在默认任务列表结果中。
- 本 PRD 的范围仅覆盖“任务列表页 v0”，不自动扩展为完整任务流系统 PRD。
