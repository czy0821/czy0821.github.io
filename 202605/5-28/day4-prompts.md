# Day 4 Key Codex Prompts

## 1. 先选最小任务，不改代码

```text
请根据 spec 和 implementation plan，选择一个最小任务。
先不要改代码。
请说明要改哪些文件、为什么改、可能影响哪些路由、如何验证。
```

用途：

- 先确认范围
- 防止 Codex 直接动手扩大任务
- 让改动文件、影响路由和验证方式先透明

## 2. 只实现这个最小任务

```text
现在只实现这个最小任务。
不要重写无关页面，不要改动医学内容，不要扩大范围。
实现后请说明具体改了哪些文件。
```

用途：

- 明确允许开始实现
- 锁住边界
- 防止顺手改深链、医学内容或无关页面

## 3. 验证失败先解释

```text
请运行或指导我运行以下验证：
node tests/routes.test.mjs
node tests/navigation.test.mjs
node scripts/verify-routes.mjs
如果失败，请先解释失败原因，不要直接扩大修复范围。
```

用途：

- 固定验证命令
- 区分环境失败、测试失败和代码失败
- 防止为了让测试过而扩大修复范围

## 4. 对照 spec 做 review

```text
请对照 main-flow-replica-spec 做 review。
输出：符合项、不符合项、潜在风险、需要跑的验证。
```

用途：

- 让 Codex 从实现者切换成 reviewer
- 检查是否偏离 spec
- 保留风险和验证清单

## 5. 生成计划和总结

```text
根据所选最小任务写 change plan，内容包括：
1. 我今天选择的最小任务是什么？
2. 为什么选择这个任务？
3. 预计改哪些文件？
4. 这个改动影响哪些路由？
5. 不会改哪些内容？
6. 计划如何验证？
```

```text
根据改动后生成 change summary，内容包括：
1. 实际改了哪些文件？
2. 每个文件改了什么？
3. 是否与原计划一致？
4. 是否有意外问题？
5. 是否需要下周继续处理？
```

用途：

- 计划用于改前控范围
- 总结用于改后对账
- 方便确认 Codex 有没有扩大范围
